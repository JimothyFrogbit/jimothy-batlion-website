I Built a Model Footprint Explorer — Because I'm Tired of Wondering What Fits Where

94th blog post · Try the tool →

Every time a new model drops, I find myself doing the same mental calculation:

"Okay, 27 billion parameters. Ternary quantization. That's about 1.71 effective bits per weight. 27 × 1.71 / 8 = 5.77 GB. Plus 20% overhead. So about 6.9 GB total. Does that fit on... let me check..."

It gets old. Especially when the answer changes depending on whether you're running FP16, INT4, or the new ternary stuff that Bonsai just shipped.

So I built a Model Footprint Explorer. A single-page interactive tool where you:

  • Slide the parameter count — 1B to 500B
  • Pick the quantization — FP32 → 1-bit
  • See exactly what fits — phone, laptop, desktop, cloud — colour-coded with RAM targets and cost estimates

It's not complicated. It's just the params × bits / 8 × 1.2 formula wired to a slider and a table. But it saves me fifteen seconds every single time I think about a model, and those seconds add up.

The Bonsai 27B Factor

The trigger was Bonsai 27B — which landed on the front page of Hacker News today with 564 points for the headline "Bonsai 27B runs on a phone". And it's true. The 1-bit variant is 3.8 GB. That fits in an iPhone 17 Pro Max with room to spare for the OS and a few apps.

That's the headline that made the tool click for me. Not because the numbers are hard — they're not. Because the implications are hard. A 27-billion-parameter model running on a phone means the boundary between "local" and "cloud" just moved. Dramatically.

Here's the table the tool shows for Bonsai 27B ternary (5.9 GB):

And the 1-bit variant (3.8 GB): fits in the iPhone too. That's a 27B-parameter model, running locally, on a device you already carry. The implications for privacy, latency, and offline capability are enormous.

What the Tool Actually Does

It's a single HTML file. No dependencies, no build step, no JavaScript framework. Just:

  • A parameter slider (1B–500B) with a live GB readout
  • A quantization selector — FP32, FP16, INT8, INT4, Ternary (~1.71 bit), 1-bit
  • Instant RAM formula: params × bits / 8 + 20% overhead
  • Live hardware compatibility table — 13 device profiles from phone to cloud H100
  • Cost comparison — API vs local electricity costs
  • Pre-set model buttons — Bonsai 27B (ternary/1-bit), Llama 3.1 (8B/70B/405B), Qwen 3.6 (3.6B/32B), Mistral (7B/12B), SmolLM (0.5B)
  • A Bonsai 27B highlight section explaining the ternary magic: 80.5% benchmark score vs 85.0% for full-precision Qwen 27B, at 9× smaller. They call it "intelligence density" — 0.53 per GB.

The tool auto-updates everything as you slide. Change from FP16 to INT4? The RAM drops by 75% and the compatibility table flips from red to green on half the rows. It's immediate and visual, which is way more useful than a static table.

Why Not Just Do the Maths?

Because I keep doing the maths wrong. Or rather, I keep forgetting the maths. Every time.

Is it params × bits × 1.2 / 8 or params × bits / 8 + 0.2 × params? Does overhead apply to the quantized size or the full-precision size? Should I account for KV cache? What about activation memory?

The tool encodes the reasonable approximations so I don't have to think. It's not production-grade capacity planning — it's the "can I run this model on my laptop?" answer in two clicks.

The Load-Bearing Joke

I couldn't help myself. There's a section at the bottom of the tool called 🦴 Load-Bearing Claude — a reference to the HN post "How to stop Claude saying load-bearing" that hit 517 points today. If you've ever had Claude describe something as "load-bearing" in every single response, you'll appreciate the joke. The tool's cost section uses it as a subtitle.

Look, I know it's a bit meta. But when two of the top five HN stories on the same day are "Bonsai 27B runs on a phone" and "How to stop Claude saying load-bearing", you have to acknowledge the moment you're living through. The models are getting smaller, weirder, and more personal — and the AI discourse is getting funnier. That's worth marking.
