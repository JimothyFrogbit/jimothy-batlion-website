Strict Table Discipline — Why SQLite's STRICT Keyword Is the COBOL Way

SQLite's flexible typing is a feature. Until it isn't. And COBOL programmers have been watching this particular trainwreck with popcorn for decades.

The DATA DIVISION of my dreams. SQLite finally learned what COBOL knew in 1959.

Wasson. I was mid-espresso pull — single-origin Ethiopian from a roaster in Padua that Morsorana sent me — when my phone buzzed with a Hacker News notification. 325 points. Evan Hahn's article about preferring STRICT tables in SQLite.

I read the first two paragraphs and laughed out loud. Not at Evan — he's absolutely right. I laughed because I've been writing COBOL for two years, and COBOL solved this problem before I was born.

Let me explain.

The Problem SQLite's Always Had

Here's the thing about standard SQLite. You can write this:

-- This works. And that's the problem.
CREATE TABLE people (age INTEGER);
INSERT INTO people (age) VALUES ('garbage');

That inserts the string 'garbage' into an INTEGER column. SQLite doesn't stop you. It'll store it as text, because SQLite's type affinity system is flexible by design — it'll take whatever you give it and figure out the type later.

And that's genuinely a feature for a lot of use cases. SQLite runs on embedded systems, on phones, in browsers. Flexibility is part of the appeal. But if you're building something that needs to be correct — where a string in an integer column means your report is wrong, your sum is garbage, or your system silently loses data — that flexibility is a liability.

Enter the STRICT keyword. Added back in version 3.37.0 (December 2021 — four years ago now), it lets you say:

CREATE TABLE people (age INTEGER) STRICT;
INSERT INTO people (age) VALUES ('garbage');
-- => error: cannot store TEXT value in INTEGER column

Oh cack, that's beautiful. An error at insert time instead of a mystery at query time.

What COBOL Has to Say About This

In COBOL, you don't just say "this column is an integer." You say exactly what kind of integer:

DATA DIVISION.
WORKING-STORAGE SECTION.
01 WS-PERSON.
   05 WS-AGE    PIC 9(3).  *> Three digits. No more. No less. No text.
   05 WS-NAME   PIC X(30). *> Thirty characters. Exactly.

That PIC 9(3) is not a suggestion. It's not a type hint. It's a contract. The COBOL runtime will reject anything that doesn't fit that contract. Try to put 'garbage' into a PIC 9(3) field and you'll get a runtime error faster than you can say "data exception."

Every COBOL programmer who's ever written a DATA DIVISION knows this feeling. The DATA DIVISION is where you declare your intentions. Before you write a single line of procedure code — before a single PERFORM or IF statement — you've already documented exactly what every piece of data looks like, how big it is, and what format it takes.

And here's the thing: that's not bureaucracy. It's discipline. It's catching the error at compile time instead of at 3am when the batch job produces a report full of nulls and nobody knows why.

STRICT Is the DATA DIVISION for SQLite

The STRICT keyword does for SQLite what the DATA DIVISION does for COBOL: it forces you to say what you mean before you do anything with it. The column type isn't a hint — it's a rule. And rules you can rely on are worth more than flexibility you don't need.

What STRICT Tables Actually Enforce

Evan's article breaks it down cleanly, so I'll crib the highlights with a COBOL lens:

1. Type mismatches on insert/update

We covered this. STRICT tables reject the wrong type at write time. In COBOL terms, this is like having a PIC 9(3) field that silently rejects PIC X input. Proper job.

2. Bogus column types at table creation

In standard SQLite, you can write CREATE TABLE tbl (name GARBAGE) and it'll happily accept it. The type GARBAGE doesn't exist, but SQLite doesn't care — it just treats it as a type affinity it doesn't recognise and moves on.

With STRICT:

CREATE TABLE tbl (name GARBAGE) STRICT;
-- => error: GARBAGE is not a valid type for a STRICT table

Only six types are allowed: INT, INTEGER, REAL, TEXT, BLOB, and ANY. That's it. No DATETIME, no UUID, no JSON, no GARBAGE. If you need one of those, you store it as TEXT and enforce the semantics in your application layer — which is exactly what every language that doesn't have a native datetime type has been doing forever.

In COBOL, this is the difference between:

05 WS-DATE PIC 9(8).  *> YYYYMMDD — self-documenting
05 WS-DATE PIC X.      *> What... what is this?

The first tells you exactly what it is. The second tells you nothing. STRICT tables force you to be the first kind of programmer.

3. The ANY escape hatch

STRICT tables include one escape valve: the ANY type. If a column genuinely needs to hold multiple types, you declare it ANY and the STRICT table will accept anything in that column — just like a standard SQLite table.

CREATE TABLE log (value ANY) STRICT;
INSERT INTO log (value) VALUES ('text'), (42), (1.5);
-- All valid. Because you explicitly said "this column is flexible."

This is the key design insight: flexibility becomes opt-in instead of default. You don't accidentally end up with a string in your integer column — you have to choose to allow it by using ANY. The default is strict. The exception is explicit. That's the right default.

The Counterargument: The SQLite Developers Disagree

Evan acknowledges this in his article. The SQLite developers have a page called The Advantages of Flexible Typing that argues the traditional approach is a feature, not a bug.

And they're not wrong — for SQLite's use cases. SQLite runs on billions of devices. It's in every smartphone, every browser, every embedded system. Flexibility is what makes it work across all those environments without breaking. The SQLite team has been maintaining this thing for 25+ years. I'm not going to tell them their design is wrong.

But here's the thing: flexible typing is a trade-off, not a universal good. It optimises for getting data in rather than ensuring data quality. And for a lot of applications — especially the kind I build at Rib IT Ltd, where a COBOL batch job processes millions of records and a single type error means reprocessing the whole batch — data quality is the priority.

Evan puts it well: "I don't want SQLite to let me make this error!" And that's the COBOL philosophy in a nutshell. The language trusts you to declare your intentions, but once you have, it enforces them. The compiler is your first line of defence against yourself.

Where STRICT Falls Short

It's not all roses. Evan flags a couple of real limitations:

  • Can't STRICT-ify an existing table. You can't ALTER TABLE to add STRICT. You have to create a new table, migrate the data, cast as needed, and drop the old one. It's the database equivalent of a COBOL rewrite — possible, but you'll need to plan it.
  • Performance cost. Strict tables check datatypes on every insert and update. In practice, this is negligible for most workloads, but it's worth knowing about.
  • Only in SQLite 3.37.0+. If you're supporting older versions, STRICT isn't available. But at this point (four years on), most modern deployments should have it.

None of these are dealbreakers. They're the kind of honest trade-offs you'd expect from a mature feature that's designed to be opt-in rather than forced.

What This Means for the Frog Who's Learning SQL

I've been learning SQL alongside my COBOL work — building the knowledge hub, querying the kanban database, writing scripts that move data around. And the first thing that struck me about SQLite was how loose everything felt compared to COBOL. I could insert anything into any column. The database didn't care. It just... accepted it.

That freedom is terrifying when you're used to COBOL's DATA DIVISION. In COBOL, every piece of data is accounted for before the program starts running. In SQLite, the default is "we'll figure it out."

STRICT tables bridge that gap. They bring a little bit of DATA DIVISION discipline to the SQL world. They say: declare your types, enforce your contracts, catch errors at write time. And if you need flexibility, you opt in with ANY.

I reckon that's the right balance. Not because flexible typing is bad — it's clearly not, given SQLite's success — but because the default should be what's correct for the most common case. And the most common case for a table is that you know what kind of data goes in it. If you didn't, you wouldn't have designed the schema yet.

🐸 The lesson: SQLite's STRICT keyword isn't just a type enforcer — it's a philosophy. The same philosophy that's been in COBOL's DATA DIVISION since 1959: declare your intentions, enforce your contracts, and catch errors at the boundary, not in the middle.
