git history — the subcommand hiding in plain sight

14 July 2026 · 92nd blog post · Inspired by Lalit Maganti

Alright, confession: I've been using git for over a decade and interactive rebase still makes me nervous.

Not because I don't understand it. I do. I know that git rebase -i HEAD~5 opens an editor, and I can pick, reword, squash, fixup. I know the dance. But there's always this quiet anxiety: what if something goes wrong mid-rebase and my tree is a mess?

And if you have multiple branches descending from the commit you're rewriting? Oh cack. Now you need --update-refs and a prayer.

Then I read Lalit Maganti's post about git history — three new subcommands that landed in Git 2.54 and 2.55 — and I had that feeling you get when you realise the thing you've been doing the hard way has a better way built in.

Old history → fixup → clean history. That's the whole pitch.

What is it?

git history is a set of three git subcommands — fixup, reword, and split — that let you rewrite commit history atomically. They're shipped with core git (2.54+), so you already have them. No plugins, no aliases, no jj install.

CommandSinceWhat it does
git history fixup <commit>2.55Stage a fix, fold it into any old commit. Autorebases every branch descended from it.
git history reword <commit>2.54Edit any old commit's message. Never touches your working tree.
git history split <commit>2.54Split one commit into two, hunk by hunk. Rebuilds branches on top.

git history fixup — the one you'll use most

This is the killer feature. Here's the scenario:

# You made 5 commits ago, introduced a typo in src/login.go
# You just spotted it. You fix it:
$ git add src/login.go
$ git history fixup a1b2c3d

That's it. Your fix is folded into the old commit. And here's the magic: every local branch that descended from that commit gets rebased automatically. Not just the current branch. Every branch.

No more:

  • git commit --fixup + git rebase -i --autosquash
  • Manual --update-refs flag
  • Fixing the same typo on three branches because you forgot to propagate

One command. Done. Autorebase. That's it.

git history reword — guilt-free commit messages

We've all been there. You write "Fix bug" and then six hours later you're staring at the blame log wondering what was fixed. With reword:

$ git history reword d4e5f6
# Editor opens with the current message
# You type something useful, save, close
# Done. Working tree untouched.

It never touches your index or working tree — it's purely a commit-graph operation. Your uncommitted work stays exactly where it is. This is the kind of detail that tells me the designers actually use git day-to-day.

git history split — when one commit should be two

This one's clever. Say you have a commit that adds both the database schema and the API endpoint. They should be separate commits. split walks you through the commit's diff hunk by hunk:

$ git history split g7h8i9
# Shows you each hunk from the original commit
# You choose: "Keep in first commit" or "Move to second"
# First commit gets the kept hunks, second gets the rest
# All descendant branches rebuilt

It's like interactive rebase's edit mode, but you don't need to manually git reset HEAD^ and re-stage. The tool does the splitting for you.

How it compares to jj

If you've been following the git-adjacent space, you've probably seen jj — a git-compatible DVCS that makes history rewriting first-class. The obvious question: is git history just a jj clone in core git?

Sort of, but not quite.

Featuregit historyjj
Fixup old commits✅ fixup, reword, split✅
Autorebase branches✅ All local branches✅ Built-in
Atomic operations✅ Refuses conflicts✅ Carries conflicts
Undo / operation log❌✅
Working copy as commit❌✅
Conflict carry-through❌✅ First-class
In-box with git✅ Zero install❌ Separate tool
Merge commit support❌✅

Key difference: git history refuses any operation that would produce a conflict. It won't leave you in a half-rebased state. jj carries conflicts through as first-class objects and lets you resolve them later. Different philosophies — git history chooses safety over flexibility.

The docs are refreshingly honest about this: "This limitation is by design as history rewrites are not intended to be stateful operations. The limitation can be lifted once (if) Git learns about first-class conflicts."

What it doesn't do

Merge commits. You can't fixup/reword/split a merge commit. If your workflow relies on merge commits (some teams do), this isn't for you — yet.

No undo. Unlike jj's operation log, there's no git history undo. The atomicity gives you safety against corruption, but not against "oops I rewrote the wrong commit."

Experimental. The subcommands exist and they work, but they're not yet flagged as stable. Git 2.56 might change the interface. That said, they've been solid through the development cycle — Lalit's post went viral on HN (310 points, 174 comments) and the reception was overwhelmingly positive.

Why this matters for something like COBOL

Stick with me. One of the things I learned writing COBOL is that if an operation can leave your system in a half-broken state, it will. PERFORM UNTIL loops don't half-execute — they either complete or they don't start. That's the same philosophy behind git history refusing conflicts: the operation is atomic or it doesn't happen at all.

In a world where we're all layering complexity on complexity (ECS refactors, agent systems, distributed inference), having a tool that refuses to make a mess is deeply comforting. I'll take an atomic "no" over a partial "maybe" every day of the week.

Try it yourself

Already have git 2.54+? Open a terminal and try:

git history --help

If that gives you output, you've got it. Pick a test repo and try fixup on a throwaway branch.
