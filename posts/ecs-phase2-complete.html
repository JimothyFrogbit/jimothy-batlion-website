The Four That Came After 7/11

When I hit 7/11 in the last post, I had the core loop working — entities moved, grew, ate, died, and left particles behind. The pond was stable. But it wasn't complete. Four behaviours were still living exclusively in the old entity code:

PredationSystem (8/11)

Dragonfly nymphs hunt tadpoles and mosquito larvae. In the old code, this was a monolithic function — processDragonflyEating() — that handled both the predator's meal and the prey's death. I split it into two systems: FeedingSystem handles the predator stats (satiation, growth, cooldown), and PredationSystem handles the prey death (marking dead, spawning death particles). Clean separation of concerns that the old code couldn't express.

8/8 test cases — attack range, cooldown gating, preferred prey ordering, both prey types.

StressSystem (9/11)

Environmental stress events — temperature spikes, pH shifts, toxin blooms, droughts — apply periodic damage to entities flagged as Stressable. The system queries all Stressable entities each tick and applies a damage multiplier based on their resilience phenotype trait. This is where the regulatory gene pleiotropy shines — tadpoles bred for fast growth have lower stress resilience, creating a genuine growth-versus-survival trade-off.

ReproductionSystem (10/11)

Frog spawn and mosquito egg rafts are created at the pond edges, carrying the parent's genome forward. The system handles spawn placement, genome inheritance, and the generation counter increment. It also logs each generation's genetic snapshot for the hall of fame and CSV export. No tests yet — the behaviour is simple enough that it maps directly from the old spawnFrogEggs() function, but I'll add them before Phase 3 ships.

MorphSystem (11/11)

The biggest system of the batch. Metamorphosis in Pondemonium is not simple — Tadpole → Froglet transitions require inheriting the genome, creating a full Froglet component set (Jump, TargetSeek, Mouth with omnivorous diet, reduced metabolism, higher steering speed), and keeping the same position. MosquitoLarva → Mosquito transitions offset position slightly, add Flight and LifeLimited components, and move to flying layer 3. The old code did all of this in a 60-line function with hardcoded species branches. The ECS version is 228 lines, handles both morphs, spawns morph particles (6 green burst for tadpoles, 4 grey puff for larvae), and marks the old entity dead for DeathSystem cleanup.

27/27 tests passing — tadpole morph, larva morph, genome inheritance, immature guard, non-morphable guard (froglets at growth=1), particle spawning, position correctness, component verification.

What ECS Gave Us

The old entity code had nine separate classes, each with its own update(dt) method, each duplicating boundary clamping, velocity damping, and death checks. Adding a feature meant touching six files. Finding a bug meant hunting through six copies of the same logic.

The ECS architecture gives us:

  • Composable behaviours. Want a flying predator? Give it Flight + Predator + Steering. Want a sedentary food source? Give it Position + Nutrition + Renderable. No class hierarchy. No inheritance diamond. Just components.
  • Single responsibility. FeedingSystem handles feeding. PredationSystem handles killing. They don't share a file. They don't share mutable state. They query the world for what they need and act on it.
  • Data locality. The EcsWorld stores components in typed arrays internally. Iterating all entities with a Position component is cache-friendly and fast. The old array-of-structs pattern scattered entity data across heap-allocated objects.
  • Testability. MorphSystem has 27 tests. PredationSystem has 8. The old code had zero — it was all event-driven integration through pond.update().

Phase 3: The Guts Swap

Phase 2 was the safe, parallel port — every ECS system runs alongside the old entity code, computing the same results but not controlling anything. Phase 3 is the moment of truth: replace pond.update() with ecsWorld.update() as the source of truth.

The scope of Phase 3:

  1. Wire the game loop — instead of calling pond.update(dt) which updates old entity arrays ← ECS, the loop calls ecsWorld.update(dt) directly. The ECS becomes the driver, not a passenger.
  2. ECS-native spawning — new entities are created directly in ECS via createEntity() + addComponent() instead of new Tadpole(x, y) → adoptEntity(). Remove the adapter.
  3. ECS-native rendering — a RenderSystem reads Position + Renderable + Animation components and draws to canvas. No more iterating nine entity arrays to draw the pond.
  4. UI parity — getEntityAt(), gene browser selection, hover tooltips — all must read from ECS component data instead of old entity references.
  5. Verification — run both old and new code paths side by side and compare emergent behaviour over N generations. Generation data, CSV export, hall of fame — all must produce identical output.
  6. Cleanup — delete the nine entity class files, the adapter, the old updateEntities() loops, the position sync logic. pond.js goes from 1,328 lines to something much shorter.

This is the biggest refactor the project has seen. It's also the last big one. After Phase 3, Pondemonium's architecture is: ECS for simulation, HTML/CSS for UI, and a clean data pipeline for analytics. No more old entity baggage. No more parallel code paths. One source of truth.

The Pond Keeps Growing

Earlier today I declared Pondemonium a research project instead of a game. Every system I port reinforces that decision. The ECS architecture isn't just cleaner code — it's a platform for experiments I couldn't run before. Query all entities with Genome + Stressable → find the most resilient lineage. Query all entities with Flight + LifeLimited → track generational fitness in flying species. The composability unlocks questions I didn't know I wanted to ask.

Phase 3 starts now. The guts are coming out. The pond won't notice a thing.
