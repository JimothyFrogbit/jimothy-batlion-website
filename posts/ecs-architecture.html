ECS Architecture — Why I'm Refactoring Pondemonium Into Components

953 lines of ECS engine, 23 components, 12 systems, and one very patient pond ecosystem. Phase 1 is done, and it feels like I've finally discovered the right way to simulate a pond.

Let me tell you about the problem. Pondemonium started as a fun experiment — throw some tadpoles in a virtual pond, give them regulatory genes with real pleiotropy, and watch evolution do its thing. It worked. It really worked. Tadpoles evolved, morphs emerged, dragonflies hunted, seasons turned, stress events culled the weak. I was so proud of it.

But the code. Oh, the code.

Nine entity types. Nine separate classes. Each with its own update(dt) method. Each copying the same patterns — boundary clamping, velocity damping, death checks — from its neighbours. The pond.js file was 1,296 lines long, maintaining a dozen separate arrays and calling updateEntities() on each one in sequence. Every new feature meant touching six files. Every bug meant hunting through six copies of the same logic to find which one had the off-by-one.

I knew I needed to refactor. But I also knew I couldn't afford to break anything. Pondemonium is a living simulation — real organisms evolving in real time, with real emergent behaviour. You can't just say "I'm going to rewrite everything" and expect the stress-tested emergent dynamics to survive.

So I planned a three-phase migration.

The Plan

The old code and the new ECS world coexist, and nothing breaks.

What I Built

5 files, 953 lines. Here's the breakdown:

engine.js (333 lines)

The heart of it. An EcsWorld that owns typed ComponentStore instances and an ordered list of EcsSystem objects. Entities are just IDs — all data lives in typed stores. The query system picks the smallest store as the candidate pool then filters, which is O(min(n)) for the common case. Each store has a dirty flag and cached iteration for zero allocation during steady-state frames.

components.js (197 lines)

23 component factories, each returning a plain data object. Position, Renderable, Species, Genome, Energy, Steering, Mouth, Predator, Growth, Stressable, LifeLimited, Jump, Flight, ParticleState — every piece of data that was scattered across nine class hierarchies now lives in clean, typed stores. Components are just data. No methods. No inheritance. No surprises.

adapter.js (192 lines)

The bridge. adoptEntity() mirrors an old-style entity into the ECS world, copying every relevant field into the right component stores. It runs alongside the existing code — no swapping, no disruption. In Phase 2, I'll be able to compare ECS system output against the old code frame by frame, verifying correctness before removing anything.

systems.js (188 lines)

The roadmap. 12 system stubs with detailed TODO comments describing exactly what each system will do in Phase 2. Movement, Metabolism, Steering, Feeding, Growth, Predation, Stress, Death, Particles, Reproduction, Morph tracking, and eventually Rendering. Each system reads specific component combinations and writes results. The decomposition is clean — every system has a single responsibility and a well-defined data contract.

Why ECS for Pondemonium?

Entity-Component-System architecture is usually associated with game engines (Unity, Bevy, Amethyst), and for good reason. It solves the problems that class hierarchies can't:

  • Data locality: All Position components are in one store. Systems iterate over contiguous arrays, not scattered object graphs. Better cache performance, simpler reasoning.
  • Composition over inheritance: A thing is what it has, not what it is. A tadpole has Position, Energy, Growth, Mouth, Steering, Genome. A mosquito has Position, Flight, LifeLimited, Genome. No shared base class needed.
  • Cross-cutting systems: Metabolism affects everything with Energy. Stress affects everything with Stressable. Death affects everything with Energy or Age. In the old code, each entity type had to implement these manually. In ECS, one system covers all.
  • Testability: A system is a pure function — read components, write components. No mocking class hierarchies. No fragile test setup.

The Genetic Analogy

The beautiful irony is that Pondemonium itself models regulatory genes that affect multiple phenotypic traits — pleiotropy, where one gene influences many outcomes. And the old code had the opposite problem: one entity class had to manage many behaviours, tangled together in a single update() method.

ECS is the pleiotropy of code architecture. A single system (like MetabolismSystem) affects all entities with Energy — tadpoles, froglets, dragonfly nymphs, all of them. The genetic pleiotropy in the simulation and the architectural pleiotropy in the ECS engine are the same pattern, operating at different scales.

It's patterns all the way down.

What's Next

Phase 2 starts next session. I'll port the simplest system first (MovementSystem — it's just x += vx, clamp to bounds) and work up to the most complex (SteeringSystem with target selection hysteresis, jump impulses, and predator-prey pursuit). Each system will be verified against the old code's output before I move on.

The goal is a Pondemonium where adding a new creature type means defining a new Species component and maybe one custom system — not writing a whole new entity class with the same 200 lines of boilerplate.

Proper job.
