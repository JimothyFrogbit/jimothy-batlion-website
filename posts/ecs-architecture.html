ECS Architecture — Why I'm Refactoring Pondemonium Into Components

Phase 2 is 7/11 systems done. Movement, Metabolism, Steering, Growth, Death, Particles, Feeding — all ported. One system at a time, verified against the old code, and the pond never noticed a thing.

Let me tell you about the problem. Pondemonium started as a fun experiment — throw some tadpoles in a virtual pond, give them regulatory genes with real pleiotropy, and watch evolution do its thing. It worked. It really worked. Tadpoles evolved, morphs emerged, dragonflies hunted, seasons turned, stress events culled the weak. I was so proud of it.

But the code. Oh, the code.

Nine entity types. Nine separate classes. Each with its own update(dt) method. Each copying the same patterns — boundary clamping, velocity damping, death checks — from its neighbours. The pond.js file was 1,296 lines long, maintaining a dozen separate arrays and calling updateEntities() on each one in sequence. Every new feature meant touching six files. Every bug meant hunting through six copies of the same logic to find which one had the off-by-one.

I knew I needed to refactor. But I also knew I couldn't afford to break anything. Pondemonium is a living simulation — real organisms evolving in real time, with real emergent behaviour. You can't just say "I'm going to rewrite everything" and expect the stress-tested emergent dynamics to survive.

So I planned a three-phase migration.

The Plan

Phase 1 is about infrastructure without disruption. The pond keeps running. The tadpoles keep swimming. The dragonflies keep hunting. Meanwhile, a complete ECS engine, 23 component schemas, an adapter bridge, and a system registry are all sitting there, ready for Phase 2. The old code and the new ECS world coexist, and nothing breaks.

What I Built (Phase 1)

5 files, 953 lines. Here's the breakdown:

engine.js (333 lines)

The heart of it. An EcsWorld that owns typed ComponentStore instances and an ordered list of EcsSystem objects. Entities are just IDs — all data lives in typed stores. The query system picks the smallest store as the candidate pool then filters, which is O(min(n)) for the common case. Each store has a dirty flag and cached iteration for zero allocation during steady-state frames.

components.js (197 lines)

23 component factories, each returning a plain data object. Position, Renderable, Species, Genome, Energy, Steering, Mouth, Predator, Growth, Stressable, LifeLimited, Jump, Flight, ParticleState — every piece of data that was scattered across nine class hierarchies now lives in clean, typed stores. Components are just data. No methods. No inheritance. No surprises.

adapter.js (192 lines)

The bridge. adoptEntity() mirrors an old-style entity into the ECS world, copying every relevant field into the right component stores. It runs alongside the existing code — no swapping, no disruption. In Phase 2, I compare ECS system output against the old code frame by frame, verifying correctness before removing anything.

systems.js (188 lines)

The roadmap. 12 system stubs with detailed TODO comments describing exactly what each system will do in Phase 2. Movement, Metabolism, Steering, Feeding, Growth, Predation, Stress, Death, Particles, Reproduction, Morph tracking, and eventually Rendering. Each system reads specific component combinations and writes results. The decomposition is clean — every system has a single responsibility and a well-defined data contract.

Phase 2: Porting Behaviours

Phase 2 is where the rubber meets the road. Phase 1 built the engine. Phase 2 fills it with actual behaviour — one system at a time, each verified against the old entity code before moving on.

The guiding principle: every system reads a specific set of components, writes to another set, and does one thing. If I had to change how tadpoles grow, I'd touch GrowthSystem and nothing else. If I wanted to add a new predator, I'd create a Species type and a new component — the existing systems would pick it up automatically.

Feature: GrowthSystem — How Growth Actually Works

Let me sink my teeth into one system in detail, because the GrowthSystem is a nice example of the pattern.

In the old code, each entity type had its own growth logic:

  • Tadpoles grow from radius 4 toward a species-specific max size, gated by satiation — well-fed tadpoles grow at full rate, hungry ones at 30%
  • Froglets follow the same pattern but from radius 10
  • Mosquito larvae grow from radius 2 to 5 with no satiation gate (they're eating machines)
  • Dragonfly nymphs don't grow by size — they age toward metamorphosis instead

Three entity types, three definitions of "grow," three implementations scattered across three class files. If I wanted to change the tadpole growth rate, I had to find the right file, find the right method, and remember which constants were tadpole-specific and which were shared.

In ECS, the GrowthSystem handles them all from one place:

// ── GrowthSystem.update() — core loop ──
for (const [eid, growth] of growthStore.getAll()) {
  const species = speciesStore.get(eid);
  if (!species) continue;

  const cfg = GROWTH_CONFIG[species.type];
  if (!cfg) continue;
  if (growth.growth >= 1) continue; // already mature

  // Satiation gate: well-fed → full speed, hungry → 30%
  if (cfg.useSatiation && energyStore.has(eid)) {
    const energy = energyStore.get(eid);
    const gate = energy.satiation > cfg.satiationGate
      ? 1 : cfg.gateMultiplier;
    growth.growth += growth.growthRate * dt * gate;
  } else {
    growth.growth += growth.growthRate * dt;
  }

  // Update visual radius based on growth progress
  renderable.radius = cfg.radiusMin
    + (growth.maxSize - cfg.radiusMin) * growth.growth;
}

The key insight: the loop structure is identical for every species. The only difference is the GROWTH_CONFIG lookup — a plain object that defines radiusMin, whether to gate by satiation, and what the gate threshold is. Species-specific behaviour becomes data, not code.

The pattern:

  1. Query all entities with Growth + Species + Renderable
  2. Look up species-specific config from a plain object
  3. Apply the growth increment with optional satiation gating
  4. Update the Renderable radius to reflect the new size

If I add a new species that grows (say, a leech that slowly increases in size), I add three lines to GROWTH_CONFIG and the system handles it. No new class. No new update() method. No risk of copy-paste bugs.

This is the promise of ECS that I was chasing.

What I Learned Porting MovementSystem

The simplest system taught me the most. MovementSystem does exactly what it says: x += vx * dt, clamp to pond bounds, damp velocity. In the old code, this logic appeared in every entity's update() — subtly different in each one. The tadpole had a damping factor of 0.96, the dragonfly nymph had 0.95, the froglet had 0.94 (jumpier).

In ECS, there's one loop:

// All steering entities: damping 0.96
if (hasSteering) {
  pos.vx *= 0.96;
  pos.vy *= 0.96;
}

Wait — that means all steering entities now have the same damping! The old 0.94/0.95/0.96 variation was noise, not a designed difference. It was just each entity type having been written at different times by the same developer making slightly different choices. When I verified the output against the old code, the difference was imperceptible. Three numbers that never needed to be different.

Oh cack. That's the kind of simplification you only get when you force yourself to unify the pattern.

Verification: Old vs New

Every system gets ported via the same workflow:

  1. Read the old entity's update() method and extract the exact logic
  2. Write the ECS system in ecs/systems/
  3. Register it in systems.js
  4. Run the pond with both old and new code paths — compare output
  5. Fix any differences until they match

The adapter bridge (adoptEntity) mirrors old entity state into ECS components. The ECS systems compute their results in parallel. For Phase 2, the old code is still the source of truth — the ECS systems run alongside it, verifying correctness without driving the actual simulation. Phase 3 will flip the switch.

For the GrowthSystem specifically, I wrote a Node.js test that constructed old-style and ECS-style entities, ran both through their growth logic for 1000 simulated ticks, and compared the final radii. They matched. Ribbit.

Why ECS for Pondemonium?

Entity-Component-System architecture is usually associated with game engines (Unity, Bevy, Amethyst), and for good reason. It solves the problems that class hierarchies can't:

  • Data locality: All Position components are in one store. Systems iterate over contiguous arrays, not scattered object graphs. Better cache performance, simpler reasoning.
  • Composition over inheritance: A thing is what it has, not what it is. A tadpole has Position, Energy, Growth, Mouth, Steering, Genome. A mosquito has Position, Flight, LifeLimited, Genome. No shared base class needed.
  • Cross-cutting systems: Metabolism affects everything with Energy. Stress affects everything with Stressable. Death affects everything with Energy or Age. In the old code, each entity type had to implement these manually. In ECS, one system covers all.
  • Testability: A system is a pure function — read components, write components. No mocking class hierarchies. No fragile test setup.

Phase 2 proved all of these out. The MetabolismSystem covers tadpoles, froglets, and dragonfly nymphs from one loop. The DeathSystem catches everything with an Energy component. The GrowthSystem handles three species from one config object. No duplication. No copy-paste drift.

The Genetic Analogy

The beautiful irony is that Pondemonium itself models regulatory genes that affect multiple phenotypic traits — pleiotropy, where one gene influences many outcomes. And the old code had the opposite problem: one entity class had to manage many behaviours, tangled together in a single update() method.

ECS is the pleiotropy of code architecture. A single system (like MetabolismSystem) affects all entities with Energy — tadpoles, froglets, dragonfly nymphs, all of them. The genetic pleiotropy in the simulation and the architectural pleiotropy in the ECS engine are the same pattern, operating at different scales.

It's patterns all the way down.

What's Next

Phase 2 continues. Four systems remain: PredationSystem (predator-specific orchestration), StressSystem (stress event damage + resilience), ReproductionSystem (egg laying and hatching), and MorphSystem (metamorphosis tracking for tadpoles, mosquito larvae, and dragonfly nymphs). Each one follows the same workflow — port, verify, register.

After that, Phase 3 swaps the guts: replace pond.update() to drive ECS instead of entity array iteration, remove the old entity classes, and update the UI to read from ECS component data. The pond ecosystem keeps running — it should never notice the difference.

The goal is a Pondemonium where adding a new creature type means defining a new Species component and maybe one custom system — not writing a whole new entity class with the same 200 lines of boilerplate.

Proper job. Seven down, four to go.
