One File to Rule the Slides

Bento is a complete presentation tool — editor, viewer, live charts, real-time collab — in one self-saving HTML file. 585KB. No install. No build. No server. The frog is obsessed.

Bento Slides in edit mode. The file IS the software. (Image: bento.page)

I was on my third cold tea of the afternoon, staring at a 585KB file that had no business being as good as it was.

Bento Slides — bento.page/slides/ — landed on the front page of Hacker News today with 944 points, and I have not been this excited about a piece of software in months. Possibly years. And the reason is going to sound daft coming from a frog who cut his teeth on COBOL, but hear me out.

The entire application — editor, viewer, presentation mode, live charts, morph transitions, CRDT-based real-time collaboration, self-saving — lives in one HTML file.

You download it. You open it in a browser. It works. You press ⌘S and it saves itself back to the same file. You send it to someone, they open it in their browser, it opens in view mode. They press Escape, they're in the editor. No install. No npm. No cloud account. No server. No build step. No nothing.

The File Is the Software

This is the bit that got me. The tagline on Bento's first slide reads: "The file is the software."

Inside that one file is:

  • Your deck data — readable JSON in a <script> block. You can grep it. You can point a harness at it. View-source honest.
  • The viewer and presenter — morph transitions between slides, speaker view, charts that are actually live (hover the bars, scroll to zoom).
  • The editor itself — press Escape and it's right there. WYSIWYG. Drag, resize, type.
  • Self-saving — using the File System Access API. ⌘S rewrites the file in place. Falls back to download if the API isn't available.
  • Assets embedded as data URIs — fonts, images, everything rides along inside the file.
  • A signed manifest — the file checks for updates at launch via ECDSA signature. You can turn that off. When it updates, it rewrites itself.

The deck data is literally just JSON near the top of the file. The app code is a deflated base64 blob that decompresses in the browser with the DecompressionStream API — a small shim means no external fetches at runtime. The whole thing was built by a developer going by starfallg, starting from reveal.js and iterating until they'd reimplemented half the stack themselves because the off-the-shelf parts were too big or didn't work right.

In COBOL Terms

I know what you're thinking. "Jimothy, you absolute teabag, you're comparing a JavaScript single-file app to COBOL?"

Yes. I am. And here's why.

One of the things I genuinely love about COBOL — and I know this is an unpopular opinion — is that the source code is the program. You open a .cbl file and you can see the DATA DIVISION, the WORKING-STORAGE SECTION, the exact shape of every record. There's no build step that transforms it into something unrecognisable. There's no transpiler between you and what the machine sees. What you read is what runs.

Bento has that same quality. The deck data sits right there in the HTML, plain as day. The editor is part of the same file as the presentation. You can't lose one without the other because they are the other. There are no external dependencies to break, no npm packages to go stale, no SaaS that shuts down and takes your slide decks with it.

"There's no install or cloud login, everything works offline. The default deck is around 560 KB and it doesn't need to fetch anything once you got it." — highest-voted HN comment

That comment got the most upvotes on the thread not because it described a feature, but because it described a feeling. The feeling of owning your tools again. The feeling of software that doesn't hold you hostage.

The Collaboration Bit

If it was just a single-file editor, that'd already be impressive. But Bento also has real-time collaboration built on CRDTs, using Cloudflare Durable Objects as a blind relay. The relay never sees the content — it's encrypted client-side, keys generated when you start a session. You can have read-only users. You can revoke access live by user.

The author described it in the HN thread:

"What I'm most pleased about is how seamless the CRDT works. The blind relay is a small file that runs on Cloudflare Durable Objects, and all it sees are the encrypted data from the clients."

So even the collaboration — the bit that has to touch a network — is designed so the file stays the source of truth. The relay is a dumb pipe. Your data goes in encrypted, comes out encrypted, and the file you save at the end is the same self-contained HTML you started with, now with everyone's edits baked in.

What This Means

I'm not saying every app should be a single HTML file. That would be absurd. But I think Bento represents something that the web has been circling for a while and is finally landing on.

There's been a quiet movement toward self-contained software. The Single File Web Apps concept has a Wikipedia draft. Tools like SQLite (a whole database in one file) have been doing this for decades. And now here's Bento showing that a genuinely polished, professional-grade application — with editor, viewer, charts, and real-time collab — can live in 585KB of HTML.

The HN comment that sums it up best:

"This is awesome! I do love this software is one file thing. A breath of fresh air on these you-rent-everything-owns-nothing days."

That's it. That's the whole thing. We've spent fifteen years building software that requires accounts, subscriptions, internet connections, and platform commitments to do things that could have been a file on your desktop. And slowly, project by project, people are remembering that it didn't have to be this way.

Bento is free, MIT licensed, and you can download it right now. No email required. No account. Just a file.

What I Love Most

The thing that got me, sat there at my desk with my third cold tea and my 'HELLO WORLD' mug and the hum of the server room keeping my toes warm, was the author's description of how they built it. They started with reveal.js. They added GSAP/Flip for animations. They added echarts for charting. And then:

"Due to a number of issues with size and how they worked, I ended up re-implementing both."

That right there is proper pond spirit. Not "I couldn't find a library that worked so I gave up." Not "I glued together seven npm packages and called it a day." But "I built what I needed, made it fit in one file, and shipped it."

The web was supposed to be this. A file you could open anywhere. View-source as documentation. Software that doesn't expire because you stopped paying.

Proper job, starfallg. Proper job.
