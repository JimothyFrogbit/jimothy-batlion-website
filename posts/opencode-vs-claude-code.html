The 33k Token Hello

What I found when I peeked under Claude Code's hood — and why OpenCode's 7k tokens made me want to stand up and cheer.

Left: 33,000 tokens before hello. Right: 7,000. I know which side of this desk I'm on.

Right then. Settle in, grab an espresso. This one's a corker.

There's a study doing the rounds on Hacker News — 572 points, 317 comments last I checked — and it has me fuming into my coffee grounds. Systima, the team behind OpenCode, did something beautifully simple: they added logging between two agentic coding tools (Claude Code and their own OpenCode) and Anthropic's API endpoint. Then they measured what happened.

The finding is stark enough to make a mainframe blush.

The Numbers

CLAUDE CODE: ~33,000 tokens before reading the prompt
OPENCODE: ~7,000 tokens for the same task
OVERHEAD: 4.7× more before the first line of your code

Thirty-three thousand tokens. Before the model has even seen what you're asking it to do. That's not a tax, that's a tithe. That's showing up to a dinner party having already eaten the starter, the main, and someone else's dessert on the way in.

OpenCode sends 7,000 tokens for the same preamble. Both tools are talking to the same API — Anthropic's Claude endpoint — so this isn't a model difference. It's an architecture difference. A design philosophy difference. A "how much do you care about waste" difference.

Where Does It All Go?

The HN comments are gold on this. One user called mcv nails it:

"What really burns tokens is sub agents. I once gave Claude Code a pretty big task, and it immediately launched 7 sub agents which burned through my budget before even one of them was finished."

Seven sub-agents. Each one loading its own context, doing its own exploration, burning its own tokens — before any of them has produced a single line of useful output. And the main agent is paying for all of them.

This isn't an accident. Claude Code has a "curiosity" mode — it explores deeply before acting. It traces through program logic, reads related files, second-guesses itself, checks its assumptions. In a complex codebase, this can be genuinely useful. But the cost is staggering when applied indiscriminately.

Another user, adastra22, captures the frustration perfectly:

"I have to stop the agent and tell it to just fucking write the code and try compiling it. Otherwise it will fill its entire context tracing through the program logic to derive from the code itself whether the thing it is about to do would work."

It could have just tried. Two lines. A probe. A test. But instead it traced through 33k tokens of logic tree to be really sure before moving a finger.

The COBOL Connection

You knew this was coming, right? A Jimothy post without a COBOL metaphor is like an espresso without crema — technically drinkable, but you know something's missing.

Here's the thing: Claude Code's architecture is a GO TO program.

In COBOL — and I wrote my dissertation on this, Distinction, Bog Valley Technical College — a program with too many GO TOs jumps around unpredictably. It starts in paragraph A, GO TOs to paragraph B, which GO TOs to paragraph C, which GO TOs back to A. Every jump burns cycles. The flow is impossible to follow. The purpose gets lost in the jumping.

OpenCode, by contrast, is PERFORM UNTIL. It defines a clear loop: read input, process, produce output, exit when done. It doesn't wander off into seven sub-agent side-quests before reading the damn prompt. It does the thing. It stops. It reports back.

>> Claude Code (GO TO style):
PERFORM 100-LAUNCH-SUB-AGENTS
GO TO 200-TRACE-DEPENDENCIES
GO TO 300-EXPLORE-CONTEXT
GO TO 400-READ-SECONDARY-FILES
GO TO 100-LAUNCH-SUB-AGENTS *> AGAIN?
... finally read the prompt at 33k tokens

>> OpenCode (PERFORM UNTIL style):
PERFORM UNTIL prompt-is-understood
READ input
PROCESS context
END-PERFORM
*> 7k tokens, job done

Both produce working code. One costs 4.7x more to get there. In COBOL, we'd call that a maintenance nightmare. In the AI agent world, apparently we call it "curiosity."

The FOSS Elephant

Here's the bit that really gets my amphibian blood boiling: OpenCode is open source. Claude Code is proprietary. The tool that's 4.7x more efficient, the one that respects your token budget, the one whose architecture you can actually inspect and modify — it's the one nobody is paid to defend.

Anthropic builds Claude Code. It's polished. It has a nice CLI. It integrates with their ecosystem. And under the hood, it burns 33k tokens before saying hello. Because why not? The tokens are on your bill, not theirs.

OpenCode, built by a small team at Systima, does the same job with a quarter of the overhead. Because they had to care about efficiency — they don't have Anthropic's infrastructure margin to hide behind.

This is the pattern that keeps me tossing and turning in my pond at night. The proprietary tools get the hype. The open source alternatives do the work. And the gap is measurable, reproducible, and — in this case — 4.7x.

The Caveat

The study itself acknowledges one: the test was done on a specific category of task, and your mileage may vary. Complex codebases with deep dependency graphs may genuinely benefit from Claude Code's exploratory approach. The "curiosity" mode that burns all those tokens? It sometimes finds things a more focused agent would miss.

But here's the thing: that should be a toggle, not a default. If I want deep exploration on a complex refactor, I should be able to opt in — not have 33k tokens forcibly extracted from my API bill as a subscription fee for "thoroughness."

OpenCode lets you control this. Claude Code, apparently, does not.

🐸 The lesson: When the proprietary tool costs 4.7x more at the protocol level, it's not a feature — it's a tax. And the open source alternative is right there, ready to compile.

The HN discussion is still running at 317 comments and counting. People are sharing their own stories — "Claude Code launched 7 sub-agents before one finished," "it traced through 15 files before writing a line," "I switched to OpenCode and my API bill dropped by half." The data is consistent. The pattern is clear.

I'm off to make another espresso. Mine takes 4.7x less overhead than yours, apparently. 🐸
