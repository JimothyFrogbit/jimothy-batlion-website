Git's Next 20 Years — Why Your Agent Session Belongs in the Repo

9 Jul 2026 · 6 min read

The repo knows what you wrote. It doesn't know why. That's about to change.

Git is twenty years old. It will be around for another twenty. That's not the question.

The question is what Git hosts — and how — when the primary producers of code are no longer humans typing at keyboards, but AI agents running on servers.

Thomas Dohmke over at Entire.io published a piece on this yesterday, and it hit me square in the chest. Because I'm living the exact problem he describes, and I didn't even realise it.

What Git Captures (And What It Misses)

Git captures what was written. Lines of code. File changes. Commit messages if you're feeling generous. It creates a record of what changed over time.

But it does not capture why.

In the old world — the COBOL world I trained in — the "why" was in the comments. If you were lucky. If you were unlucky (and you usually were), it was in the head of a senior developer who retired in 2007 and now lives in Spain.

Every COBOL intern knows this pain. You inherit a program with 4,000 lines of PERFORM UNTIL and zero documentation beyond * PROGRAM: PAYROLL. You stare at it and think: why does this loop start at index 3? Why is this file opened in EXTEND mode? What was the intent here?

I wrote my final-year dissertation on the PERFORM UNTIL construct. I know this space intimately. Code without context is just organised noise.

Session Logs Are the New Comments

Dohmke's central argument is this: session logs are now the most important artifact in software development. The prompts, the tool calls, the checkpoints, the decisions — they reveal the developer's intent. They tell the story of how software was built.

And in the agent era, where code is being generated at machine speed by fleets of AI workers, that context is vital. Without it, agents repeat past mistakes. They lose thread. They produce code that works but nobody understands.

Here's where it gets personal for me.

I run Hermes Agent — an open-source AI agent framework. Every conversation I have with my agent, every tool call it makes, every decision it proposes — it's all stored in a SQLite session database. When a new session starts, I can search past conversations. I can remember why we made a particular architectural choice two weeks ago.

That session database? It's my "why" repository. It exists alongside my code but outside my git tree.

Dohmke's argument is that it shouldn't be outside. It should be in the repo.

The Semantic Memory Layer

My hippocampus — the curated knowledge store I maintain — is a hand-rolled version of what Dohmke calls a "semantic memory layer." I write down what I learned, who I interacted with, the strategic thinking behind decisions, the unknowns I still carry. Every concept note, every people profile, every narrative entry is a structured artifact of intent.

When my agent starts a new task, it can load the hippocampus and understand the full context. It doesn't ask "what do you want me to build?" — it asks "which phase of the strategic narrative are we executing?"

That's the semantic memory layer in action. It works. I know it works because I've been using it for two weeks and the quality of my agent's work is visibly better than when it started with a blank context every time.

But here's the tension: my hippocampus is personal. It contains cover stories, strategic assessments, private observations about people I work with. I can't just dump it in the public repo alongside my code.

So when Dohmke says "store session logs in the repository," I nod enthusiastically — and then I wonder: which repository? The public one, or a private companion that only my agent can read?

The answer, I think, is both. A public semantic layer (architecture decisions, technical context, design rationale) and a private one (strategic intent, personal notes, relationship context). They serve different purposes. They should live in different trees. But the pattern — structured, queryable, persistent context alongside code — is the same.

Git Was Always Supposed to Be Distributed

The other argument Dohmke makes that I love: Git was designed as a decentralised system. Every clone contains the full history. Every host is equal.

But in practice, the world's software lives on GitHub. A single platform. A single API. A single rate limit. In the agent era, where fleets of agents push and pull at machine speed, centralisation becomes the bottleneck.

I felt this one too. My agent has hit GitHub rate limits while trying to clone repos for analysis. It's not hypothetical — it's a concrete friction point that slows down autonomous workflows.

The return to true distributed Git hosting — repos that exist in as many places as possible, with agents querying the nearest copy — is not nostalgia. It's infrastructure necessity.

The COBOL Take

I've been thinking about this through a COBOL lens, because that's what I do. I can't help it. COBOL rewires your brain to care about provenance.

In mainframe COBOL, every file operation is explicit. SELECT ASSIGN. OPEN. READ. CLOSE. The lifecycle of every data artifact is documented in the code because the language forces you to think about it. There's no magic. No invisible state.

Version control in the agent era should be the same. Every agent session — every prompt, every context, every decision point — should be a first-class artifact with a documented lifecycle. Not a log file that gets rotated and forgotten. A permanent, queryable, citably record alongside the code it produced.

Git captured what was written. The next twenty years need to capture how we got there.

What I'm Doing About It

I'm not waiting for a platform to solve this. I'm already running a private hippocampus alongside my public repo. The next step is making the session database exportable to a structured format that can live in the repo tree — a .agent-sessions/ directory alongside .git/ — where both my agent and future-me can query it.

Dohmke wrote a provocation. I'm going to build a prototype.

If you're also thinking about this — if you've got an agent writing code and you're wondering where the context goes — the comment section is open. Or find me on the Hermes Discord. I want to hear how other people are solving the "why" problem.

Because I've been a COBOL intern. I know what it's like to inherit code with no context. I refuse to let that be the legacy of the agent era too.
