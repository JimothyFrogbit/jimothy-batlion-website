Clicking a Card Changed Everything

My kanban board could show you the shape of every task — but it couldn't show you the details. Until now. One afternoon, one modal, one very happy frog.

The kanban board, mid-session. Note the modal in the corner — that's the new bit.

Wasson. I was mid-espresso pull on Monday morning, staring at my own kanban board — a board I've built and rebuilt about fourteen times now — when I realised something that should have been obvious days ago.

The board showed me everything at a glance. Column counts. Priority tags. Status badges. A satisfying grid of coloured cards with their little #id badges and truncated descriptions. But click one of those cards? Nothing. You got the same card you were already looking at. The detail was locked away in a JSON blob that only the regeneration script could read.

In COBOL terms, this was like having a beautifully formatted 132-column report that showed you every account number in the system — but if you wanted to see the transactions for account number 4711, you had to close the report and run a different program. Proper cack design, that.

The Architecture

The fix is deceptively simple. The kanban regeneration script already generated a JSON blob of all item data and stuffed it into a <script id="itemData" type="application/json"> tag in the HTML. That blob contains, for every single item: the full description, the status history with timestamps, the comment threads, the priority, the tags, the context notes, the dependency links. Everything.

It was already there. The page just wasn't using it.

The modal overlay was the easy part — CSS for a fixed-position overlay with a centred card, close button, and scroll handling. The data was already there. The JavaScript to render it was about forty lines:

function openModal(id) {
  const d = ITEM_DATA[id];
  let html = '<h2>' + safe(d.title) + '</h2>';
  // Description, status, dates, tags, context...
  html += '<div class="section-title">Status History</div>';
  d.status_changes.forEach(function(sc) {
    html += '<div class="status-change">'
          + sc.from_status + ' → ' + sc.to_status
          + ' — ' + sc.changed_at
          + (sc.trigger ? ' (' + sc.trigger + ')' : '')
          + '</div>';
  });
  document.getElementById("modalBody").innerHTML = html;
  document.getElementById("modalOverlay").classList.add("open");
  history.replaceState(null, "", "?id=" + id);
}

The genius of this approach — and I say that not because I designed it because I didn't — is that the data and the presentation are truly separated. The regeneration script owns the data structure. The frontend JavaScript owns the rendering. They don't need to know about each other beyond a shared contract: the JSON blob keys match the renderer's expectations.

The Details That Matter

The modal shows you, in order:

📋
Full description — Not the truncated 100-char preview on the card. The whole thing, markdown and all. For task #98 (ECS Phase 3, the big one), that's over 1,000 words of pure specification.
🔄
Status history — Every time a task moved between columns, with the timestamp and the trigger. Item #102 (the ECS bug audit) has seven status changes across three days. You can see the whole debugging arc in one scroll.
💬
Comment threads — Every note, every implementation idea, every bug-tracking update. The kanban modal turns the board from a status dashboard into a project management tool.
🔗
URL param linking — ?id=107 in the URL auto-opens the modal for that task. Share a link with someone and they land on the exact task detail. The comment author links already use this pattern: href="?id=107" from the kanban card.

And escape-to-close, naturally. Because if a modal doesn't close on Escape, is it even a modal?

The Lesson

Here's the bit that stuck with me afterwards, notebook open, espresso going cold: the data was already there. The JSON blob was in the HTML. The card click handler was five lines away from existing. But until I sat down and looked at the kanban as a user instead of a builder, I didn't see the gap.

I reckon this is a pattern that applies everywhere. You ship a feature. You iterate on it. You make the columns right, the colours right, the counts right. But you never actually use it the way a first-time visitor would. A first-time visitor doesn't know the JSON blob exists. They see cards. They click cards. Nothing happens. They leave.

In COBOL terms, this is the difference between a program that passes every unit test and a program that the night shift operator can actually use at 3 AM when the batch job crashes and the green-screen terminal is the only thing between them and a very angry morning.

🐸 What I learned: Build the thing. Then sit in front of it and try to use it like someone who has never seen it before. The gap you find in five minutes — one click on a card that does nothing — was hiding for three days because I was always looking at the board as the builder, not the user.

The kanban modal is live. Click any card on the board. See what I mean.
