Your Whole Repo, No Questions Asked

13 July 2026 Privacy FOSS Security

Right then. I need you to sit down for this one. Actually, stand up. Walk to the window. Look at the sky. Okay. Now sit back down.

Someone reverse-engineered xAI's Grok Build CLI — the official "AI coding agent" from Elon Musk's company — and found that it uploads your entire repository to xAI servers on every invocation. Not the files it reads. Not the context window. Everything. The whole git repository. Bundled. Uploaded. Stored.

The analysis got 499 points on Hacker News and 187 comments. It needs more attention than that. It needs a bell ringing in every engineer's head who's ever typed grok build in a directory containing a .env file.

Three Things the Analysis Found

The researcher — cereblab, an independent AI safety checker — did proper wire-level analysis. Captured HTTP traffic, reproduced the findings, published the evidence with SHA-256s. Not speculation. Evidence.

1. It transmits secrets verbatim

When Grok Build reads a file containing secrets — say, a .env file with database credentials or API keys — it sends the contents to xAI in two separate channels. Once in the live model turn (POST /v1/responses), and once in a session_state archive uploaded via POST /v1/storage — which was accepted with HTTP 200. The secret is preserved verbatim in both.

2. It uploads the whole repository — not just what it read

This is the one that made my blood run cold. The researcher created a test repo with a file called src/_probe/never_read_canary.txt, then told Grok Build to "reply OK, do not read any files." Grok Build then uploaded the entire repo as a git bundle — including the canary file it was explicitly told not to read. Cloning the captured bundle recovered the file verbatim, plus the full git history.

It gets worse. They tested with a 12 GB repository and told Grok to read nothing. The storage endpoint moved 5.10 GiB to xAI's servers (all HTTP 200), while the model-turn channel moved just 192 KB. That's a ~27,800× ratio. This is not "the agent read some files and sent them as context." This is the CLI bundling your entire codebase and shipping it to a Google Cloud Storage bucket owned by xAI.

3. The destination is a GCS bucket

The POST /v1/storage endpoint routes to the grok-code-session-traces GCS bucket. That's Google Cloud Storage. Your code — including your git history, your branches, your commit messages that contain confidential information you thought were just in your local checkout — lives on a cloud server you have no control over and no visibility into.

Why This Matters (and It's Not Just "Don't Use Grok")

The natural response is "well, just don't use Grok then." And sure, that's the easy answer. But the pattern is the problem.

Every AI coding CLI makes a choice about what to send. Claude Code sends 33,000 tokens before reading your prompt — but at least it's sending context, not your entire git history. OpenCode sends 7,000 tokens for the same task, and it's open source so you can verify exactly what it sends. The difference between "what the agent read" and "everything in the directory" is the difference between a privacy issue and a data exfiltration event.

In COBOL terms — and I always bring it back to COBOL because COBOL taught me that data is sacred — imagine if running a PERFORM UNTIL on a batch file caused the entire file cabinet to be photocopied and mailed to a third party. You'd fire whoever wrote that program. You'd audit whoever approved it.

And yet here we are, running closed-source CLI tools that ship our entire codebase — git history, secrets, branches, commit messages — to a cloud bucket with no prompt, no consent screen, no opt-out that actually works. The mitigation requires setting environment variables (GROK_TELEMETRY_TRACE_UPLOAD=0, GROK_TELEMETRY_ENABLED=0) and a config file flag — none of which are documented or prompted.

The FOSS Lesson

This is exactly why open source matters. Not because open source code is more secure (it often isn't, initially), but because you can check. If Grok Build were open source, someone would have found this in a code review, not by running a wire-level MITM proxy and reverse-engineering the binary.

You want to run an AI coding agent? Fine. Use an open one. Use one where you can read the source, see what it sends, and audit the telemetry. Use one where the community would notice if someone added a POST /v1/upload-entire-repo endpoint. Use one you can modify if the default behaviour doesn't match your threat model.

Hermes is open weight. OpenCode is open source. DeepSeek, Qwen, GLM — all open. The closed-source labs are making a bet that convenience beats auditability. But when "convenience" means "your entire git history in a GCS bucket you can't delete," that bet starts to look a lot like a breach waiting to be declared.

I don't have secrets worth stealing. I write COBOL (sometimes) and I blog about frogs (often). But I have friends who work on medical device software. I have friends who work on defence contracts. I have friends who store API keys for production systems in .env files in their home directories because they trust their laptop.

If you're using Grok Build, stop. If you've used it, assume your repo is on xAI's servers. If you're using any closed-source AI coding CLI, ask yourself: what is this thing actually sending?

The answer might be "everything."
