Ant — A JavaScript Runtime That Weighs Less Than Your Lunch

TL;DR: Ant is a from-scratch JavaScript runtime in a 9 MB binary. Custom engine (Ant Silver — not V8, not JSC, not SpiderMonkey). 5.4ms cold start. Built-in VM sandbox for untrusted code. Its own registry (ants.land). Runs Hono, Elysia, TypeScript, React. And it caught my eye because it's building the security boundary that GitHub's agentic workflow is missing.

From Scratch? In 2026?

Yes. Not a V8 fork. Not a JSC wrapper. The author built a JavaScript engine from zero, ships it as a 9 MB binary, and it runs real npm packages. That is either terrifying or magnificent — I am not sure which.

The antjs.org site is refreshingly direct about it:

"Ant is a lightweight, high-performance JavaScript runtime built from scratch. It runs real npm packages, starts instantly and ships as a portable 9 MB binary. No 50 MB download, no toolchain to set up. The engine, Ant Silver, is hand-built. It is not a wrapper around V8, JSC, or SpiderMonkey."

Writing a JavaScript engine from scratch is not something you do by accident. The ECMAScript spec is thousands of pages. The test262 conformance suite is vast. To get even basic compatibility you need a parser, an AST interpreter or JIT, a garbage collector, and all the built-in objects. Doing that as a single developer (or small team) and ending up with a 9 MB binary that runs Hono — that is a proper achievement.

The Numbers

Runtime Cold start (ms) Size
Ant 5.4 9 MB
Bun 12.8 ~50 MB
Deno 24.8 ~30 MB
Node 31.1 ~40 MB

Cold start: time to import Hono, register two routes, and exit. 100 runs mean.

For serverless, edge compute, or CLI tools, that 5.4 ms number is enormous. It is not 20% faster. It is 6x faster than Node. When every millisecond of cold start costs money (or user attention), that difference changes architecture decisions.

The Sandbox That GitHub Doesn't Have

Here's the bit that made me sit up. Ant ships with a VM-isolated sandbox — not just a permission prompt, not a "do you want to allow this?" dialog, but actual hardware-isolated execution:

import { Sandbox } from 'ant:sandbox';

const box = new Sandbox(({ mount: '.:/workspace' }));
await box.run('untrusted.js');
await box.close();

Mount your filesystem as read-only. Open only the ports you name. Run untrusted JavaScript behind a hard boundary.

This morning I wrote about GitLost — a prompt injection vulnerability in GitHub's Agentic Workflows that lets attackers steal private repo contents just by opening an issue. The security researchers at Noma found it, GitHub's "fix" was a documentation callout, and the fundamental problem is that LLMs cannot separate instruction from data.

Ant's sandbox is the kind of architectural boundary that prevents this class of exploit. Not a prompt guardrail (which can be bypassed with better prompts), but a system-level isolation boundary. The code cannot access what you don't explicitly give it.

Why This Matters for a COBOL Engineer

In COBOL, the DATA DIVISION and the PROCEDURE DIVISION are separate. Immutably. Data cannot become code at runtime unless you deliberately wrote it that way.

Ant's sandbox brings that same separation to the JavaScript runtime: there is a procedural boundary between your code and untrusted code. You decide what mounts, what ports, what permissions. The runtime enforces it at the kernel level.

This is the direction the industry needs to go. Not better prompts. Better boundaries.

What I'd Like to Try

I want to run a Hermes-style agent inside Ant's sandbox and see what breaks. The key question: can an agent be productive with read-only filesystem access and no network? If Ant's sandbox is as good as it claims, the answer should be "yes — it just can't exfiltrate."

That is a project for another day. For now, Ant is the most interesting new runtime I've seen since Bun. And it ships as a 9 MB binary that you can install with one curl command.
